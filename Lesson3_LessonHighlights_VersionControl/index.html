Highlight #1 Versioning Control Systems

When members of a team are working on the same project files coming from many different people can be a pain to manange. A versioning control system is sort of like a storeroom for all of the files in a certain project. Team members can check a certain file out of the repository and make changes to that file, whether that be adding or deleting information. After they are finished editing it then they submit the file back to the repository with the saved changes. The system can automatically note who made the changes, what changes they made, and when they edited the file. All of this is kept on record so that the entire team can see who is doing what to the project.

Highlight #2 What's Git?

Git is an open source VCS that is made specifically for projects of all sizes, and handles them all with amazing speed and efficiency. Easily downloaded off of the Git website, members can begin using the features immediately and i'm sure will be surprised by how well it implements multiple workflows, convenient staging areas, and cheap local branching. User have expressed their enjoyment with its performance and say it easily passes rivals like Subversion, Perforce, and ClearCase. Git hasn't been n the scene forever, but it quickly became the go to version control system for projects large and small.

Highlight #3 Git's History

Git was developed thanks to the unfortunate falling out of two companies. Linux is a large open source software project and for the early years of its startup file sharing was a mess. After about ten years the Linux community starting using a VCS called BitKeeper when they wanted to pass around files they recently edited. But after a disagreement that shut down the free of charge arrangement between Linux and BitKeeper, so something had to be done. That’s when Linux founder, Linus Torvalds decided that a new VCS needed to be made for their sharing needs, and taking what they learned from BitKeeper, Torvalds and the Linux community created a wicked fat and extremely efficient VCS that could handle the massive size of Linux file sharing, as well as any small project that wanted to features as well.

Highlight #4 Remote vs. Local

Versioning control systems or VCS can store files and the changes made to them in two different repositories. One of which is a local repository, this repository is located on a individuals own server, it allows the user to make changes and save them to their own machine before sending them off to be seen at the remote repositroy. Now there are two ways in which one can make a local repository, you can either make a brand new repository, or make a copy of the exisiting remote repository and save it on to your own server, thus making it local to you. The other type of repository mentioned above is a remote repository, these are file sharing storerooms that exist at a centralized server, usually a much bigger one than an individual would have at their house. This is the server that all team memebers send their edited files to for them to be merged together making one on going project file. The remote repository allows team members to push or pull specific files from the server and have access to other members work.

Highlight #5 But how does Git work?

Once all of the changes have been made to file the user then has to “commit” the file to the remote repository. This takes a few connections between your workspace and the Git server. Terminal is used to commit or push the newly edited file in the repository. The user types command that look sort of like gibberish but to a trained eye they are the exact steps it takes for files to be accessed by the entire project team. Once the master user has been establishes, the user types in “git init” to initialize the repository then add the selected files in and commit them with the command git commit –m” finally ended with “git push” to push all of the edited files onto the remote server.
